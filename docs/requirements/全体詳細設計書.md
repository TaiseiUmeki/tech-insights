# 全体詳細設計書

## 1. バックエンド詳細設計(オニオンに即す)

TechInsights のバックエンドは、FastAPI と SQLAlchemy を用いたオニオンアーキテクチャで構成する。依存方向は `presentation -> application -> domain <- infrastructure` とし、DI 層のみが全層を結合する。

### 1.2 domain層

domain 層はビジネスルールの中核であり、他層を import しない。Python 標準ライブラリと Pydantic のみを利用し、DB、FastAPI、SQLAlchemy、embedding model には依存しない。

| 要素 | 内容 |
|---|---|
| Entity | `Article`、`Category`、`Author` を Pydantic `BaseModel` で表現する |
| Value Object | `ArticleTitle`、`SearchQuery` など、単体で検証したい値を不変オブジェクトとして定義する |
| Repository IF | `ArticleRepository`、`CategoryRepository`、`AuthorRepository` を ABC として定義する |
| Exception | 記事未存在、カテゴリ未存在、著者未存在、検索条件不正などをドメイン例外として定義する |

`Article` は `id`、`source_article_id`、`category_id`、`author_id`、`title`、`content`、`embedding`、`published_at` を保持する。カテゴリ名や著者名の解決は domain 層では行わず、ID で関係を表す。

Repository IF は primitive 型を入力として受け取り、単一取得では `Optional[Entity]`、一覧取得では `items` と `total` を含む検索結果オブジェクトを返す。検索モードは `keyword`、`semantic`、`hybrid` の3種類に限定する。

### 1.3 infra層

infra 層は domain 層の Repository IF と Application Interface を実装し、DB、CSV、embedding model との接続を担当する。application 層と presentation 層は import しない。

| 要素 | 内容 |
|---|---|
| DB Model | `ArticleModel`、`CategoryModel`、`AuthorModel` を SQLAlchemy で定義する |
| Repository実装 | SQLAlchemy Session を受け取り、Model と Entity の変換を行う |
| 検索実装 | PostgreSQL Full Text Search、`pg_trgm`、pgvector HNSW を利用する |
| Embedding | `intfloat/multilingual-e5-small` をローカルで実行し、APIキー不要にする |
| CSV取込 | `docs/articles.csv` を読み込み、`source_article_id` で冪等性を担保する |

`ArticleModel.to_entity()` は ORM Model を domain の `Article` に変換する。カテゴリ名・著者名を含むレスポンス DTO は application 層で組み立てる。

キーワード検索は `websearch_to_tsquery('simple', :q)` と `to_tsvector('simple', title || ' ' || content)` を使う。`pg_trgm` はタイトル・本文の部分一致や表記ゆれ補助に使う。セマンティック検索は `articles.embedding vector(384)` を `embedding <=> query_embedding` で検索し、`1 - cosine_distance` を関連度として扱う。ハイブリッド検索は keyword と semantic の候補を RRF で順位融合する。

物理削除のみを採用し、論理削除カラムは持たない。外部キーは `ON DELETE RESTRICT` とし、CASCADE は使用しない。

### 1.4 application層

application 層はユースケースの実装と DTO 変換を担当する。domain 層のみを import し、infrastructure と presentation は import しない。

| Usecase | 責務 |
|---|---|
| `ArticleUsecase` | 記事一覧、詳細、作成、更新、削除を制御する |
| `SearchUsecase` | keyword、semantic、hybrid の検索要求を解釈し Repository に委譲する |
| `MasterUsecase` | カテゴリ一覧、著者一覧を取得する |
| `ReindexUsecase` | embedding または検索 index の再生成要求を制御する |

Presentation 層からは request schema を直接受け取らず、必要な primitive 値を引数で受け取る。戻り値は Application DTO とし、記事一覧 DTO には `items`、`page`、`limit`、`total` を含める。

記事作成・更新では、カテゴリ名と著者名を既存マスタへ解決し、存在しない場合は追加する。`title` または `content` が変わった場合は embedding を再生成する。embedding 生成に失敗した場合は保存を失敗扱いにし、古い embedding を使った検索結果を返さない。

### 1.5 presentation層

presentation 層は FastAPI の HTTP エンドポイント、Request/Response schema、エラー変換を担当する。application 層と DI 層を import し、infrastructure 層は import しない。

| API | ルーター | 用途 |
|---|---|---|
| `GET /api/articles` | `article_api.py` | 記事一覧取得、keyword/semantic/hybrid 検索 |
| `POST /api/articles` | `article_api.py` | 記事作成 |
| `GET /api/articles/{articleId}` | `article_api.py` | 記事詳細取得 |
| `PUT /api/articles/{articleId}` | `article_api.py` | 記事更新 |
| `DELETE /api/articles/{articleId}` | `article_api.py` | 記事物理削除 |
| `GET /api/articles/{articleId}/related-articles` | `article_api.py` | 関連記事取得 |
| `GET /api/categories` | `category_api.py` | カテゴリ一覧取得 |
| `GET /api/authors` | `author_api.py` | 著者一覧取得 |
| `POST /api/reindex-jobs` | `reindex_job_api.py` | 再インデックスジョブ作成 |

検索専用の別エンドポイントは作成せず、RESTful に `GET /api/articles?q={query}&searchMode={mode}` で扱う。`q` 未指定時は通常一覧、`q` 指定時は `relevance` 順を既定とする。

Response schema は Application DTO から `from_dto()` で変換する。Domain Entity を presentation 層へ直接露出しない。ドメイン例外は 400、404、422、500 などの HTTP ステータスへ変換する。

### 1.6 その他

DI 層は唯一全層を import できる結合層とする。`get_article_usecase(db: Session = Depends(get_db))` のようなファクトリ関数で Repository 実装、Embedding Service、Usecase を組み立てる。

設定値は `config.py` に集約する。embedding model は固定であるため DB カラムには保持せず、設定・README・検索設計書で管理する。

Alembic では `vector` と `pg_trgm` 拡張、`categories`、`authors`、`articles`、全文検索 index、trigram index、HNSW index を管理する。テストは domain の値検証、application のユースケース、repository のDB検索、presentation のAPI結合を分けて実施する。

### 2.2 shared層

shared 層は最下層であり、他の FSD 層を import しない。外部ライブラリ、共通 UI、共通型、HTTP client、ユーティリティを提供する。

| セグメント | 責務 |
|---|---|
| `api/` | `HttpClient` を提供し、base URL、timeout、共通エラー処理を管理する |
| `ui/shadcn/` | button、dialog、input、select、badge、pagination などの共通 UI を提供する |
| `ui/form-fields/` | TextField、TextareaField、SelectField などフォーム部品を提供する |
| `lib/` | `useEnterSearch`、APIエラー処理、グローバルフォント、汎用 hooks を提供する |
| `utils/format/` | 日付や数値の表示整形を提供する |
| `types/` | ページング、APIエラー、検索モードなどの共通型を定義する |

shared 層には Article 固有の API 型や検索ロジックを置かない。Article 固有の型は entities、検索フォームの意味づけは features に置く。

### 2.3 entities層

entities 層はバックエンド API と1対1に近い型定義・API client・query key を管理する。shared 層のみ import できる。

| Entity | 内容 |
|---|---|
| `article` | 記事一覧、詳細、作成、更新、削除、関連記事の API 型と API client |
| `category` | カテゴリ一覧 API 型と API client |
| `author` | 著者一覧 API 型と API client |

`article/model/api-types.ts` には `ArticleListResponse`、`ArticleDetailResponse`、`ArticleCreateRequest`、`ArticleUpdateRequest`、`ArticleSearchParams` を定義する。API レスポンスのフィールドはバックエンドに合わせ、`sourceArticleId`、`publishedAt`、`searchMode`、`score` などを扱う。

`article/api/article-api.ts` は `listArticles`、`getArticle`、`createArticle`、`updateArticle`、`deleteArticle`、`getRelatedArticles` を関数として提供する。検索も `listArticles(params)` に含め、専用の search API client は作らない。

`query-key.ts` は `articleQueryKey.all`、`list(params)`、`detail(articleId)`、`related(articleId, params)` のように階層化し、検索条件を key に含める。

### 2.4 features層

features 層はユーザーが実行したい操作を hook、schema、型として提供する。entities と shared のみ import でき、widgets や page-components は import しない。

| Feature | 責務 |
|---|---|
| `article/list` | 初期一覧、ページング、公開日順取得の hook を提供する |
| `article/search` | `q`、`searchMode`、カテゴリ、著者、sort を意味のある検索条件に整形する |
| `article/get` | 記事詳細取得 hook を提供する |
| `article/create` | 作成 mutation、Zod schema、フォーム型を提供する |
| `article/update` | 更新 mutation、Zod schema、フォーム型を提供する |
| `article/delete` | 削除 mutation と一覧再取得の invalidation を提供する |
| `category/list` | カテゴリ選択肢取得 hook を提供する |
| `author/list` | 著者選択肢取得 hook を提供する |

`useSearchArticles` は UI から受け取った検索入力を API パラメータに変換する。`searchMode` は `keyword`、`semantic`、`hybrid` のみ許可する。`q` が空の場合は通常一覧として扱う。

作成・更新フォームでは Zod schema で `title`、`content`、`authorName`、`categoryName`、`publishedAt` を検証する。バリデーションメッセージは日本語とする。

### 2.5 widgets層

widgets 層はユースケースに紐づく再利用可能な UI を管理する。features、entities、shared を import できるが、page-components は import しない。

| Widget | 責務 |
|---|---|
| `search-toolbar` | 検索入力、検索モード切替、カテゴリ/著者フィルタ、クリア操作を提供する |
| `article-card` | カテゴリ、タイトル、スニペット、著者、公開日時、score を表示する |
| `article-form` | 作成・編集フォームを共通化する |
| `article-detail-modal` | 記事本文、メタデータ、関連記事、編集/削除導線を表示する |
| `article-delete-dialog` | 物理削除の確認と削除実行を扱う |
| `empty-state` | 検索結果0件やAPIエラー時の表示を共通化する |

検索モード切替は segmented control とし、`keyword`、`semantic`、`hybrid` を明示する。`semantic` と `hybrid` では関連度スコアを表示し、`keyword` では一致スニペットを優先する。

1画面だけで使う大きな配置用コンポーネントは widgets には置かず、page-components の `ui-block/` に置く。

### 2.6 page-components層

page-components 層はページ固有の UI と状態配置を管理する。app 層の `page.tsx` から呼び出される Container を提供し、他の page-components は import しない。

`ArticleHomeContainer` は TechInsights のメイン画面を構成する。テンプレート由来のダッシュボード画面とサイドバーは設けず、画面上部に `TechInsights` を表示する。

| ui-block | 責務 |
|---|---|
| `search-panel` | `search-toolbar` を配置し、URL query または内部 state と同期する |
| `article-list` | 記事カードのグリッド、ページネーション、ローディング、空状態を制御する |
| `modal-area` | 詳細、作成、編集、削除確認の表示状態をまとめる |

Container はページ全体の選択状態、モーダル開閉、ページング、検索条件を管理する。データ取得そのものは features の hooks に委譲する。

### 2.7 app層

app 層は Next.js App Router のルーティング、Provider、グローバル設定を管理する。page-components を import してページとして公開する。

| ファイル | 責務 |
|---|---|
| `layout.tsx` | HTML骨格、metadata、共通フォント、グローバルProviderを設定する |
| `providers.tsx` | React Query などのクライアント Provider を設定する |
| `page.tsx` | `ArticleHomeContainer` を表示し、TechInsights の初期画面にする |
| `globals.css` | Tailwind とグローバルスタイルを定義する |

本システムでは認証を対象外とするため、記事管理画面はログイン前提にしない。既存テンプレートの authenticated layout や sidebar は TechInsights のメイン画面には利用しない。
