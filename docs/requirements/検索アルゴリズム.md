# 検索設計書

## 1. 設計方針

TechInsights の記事検索は、明確な技術用語で探すキーワード検索、自然文の意図で探すセマンティック検索、両者を組み合わせるハイブリッド検索を提供する。

精度を優先しつつ、評価者のローカル環境で再現できることを重視する。外部 API キーを必要としない固定 embedding model と PostgreSQL の検索機能を組み合わせ、検索 API は RESTful に `GET /api/articles` の一覧取得として扱う。

### 1.1 検索モード

| searchMode | 用途 | 採用方式 | 主な並び順 |
|---|---|---|---|
| keyword | 明確な技術用語、製品名、著者が分かる検索 | PostgreSQL Full Text Search + pg_trgm | キーワード関連度順 |
| semantic | 自然文や曖昧な意図から近い記事を探す検索 | pgvector による記事単位 embedding 検索 | cosine similarity 順 |
| hybrid | 用語一致と意味的類似の両方を使う検索 | keyword と semantic の RRF 融合 | RRF スコア順 |

`q` が未指定の場合は検索処理を行わず、記事一覧を `published_at DESC` で返す。

### 1.2 対象データ

- 検索対象は `articles.title` と `articles.content` とする。
- カテゴリと著者は `categoryId`、`authorId` による絞り込み条件として扱う。
- セマンティック検索は記事単位の `articles.embedding` を利用する。
- チャンク単位の検索は後続実装とし、今回の設計では `article_chunks` テーブルを追加しない。

## 2. キーワード検索

### 2.1 採用方式

キーワード検索は PostgreSQL Full Text Search を主軸とし、`pg_trgm` による類似度を補助的に利用する。

| 要素 | 採用内容 |
|---|---|
| クエリ生成 | `websearch_to_tsquery('simple', :q)` |
| 検索対象 | `to_tsvector('simple', title || ' ' || content)` |
| FTS順位 | `ts_rank_cd(search_vector, query)` |
| 補助順位 | `similarity(title, :q)`、`similarity(content, :q)` |
| インデックス | GIN expression index、trigram GIN index |

`simple` 辞書を使う理由は、`docs/articles.csv` の本文が英語中心かつ技術用語を多く含み、ステミングや言語別辞書による意図しない正規化を避けるためである。

### 2.2 スコア

keyword score は Full Text Search の順位を主にし、trigram 類似度を補助的に加点する。

```text
keyword_score = ts_rank_cd_score + max(title_trgm_score, content_trgm_score) * 0.2
```

画面では score を内部順位用の関連度として扱い、ユーザーには一致スニペットを優先表示する。

## 3. セマンティック検索

### 3.1 採用方式

セマンティック検索は `intfloat/multilingual-e5-small` を固定モデルとして利用し、記事単位 embedding を pgvector に保存する。

| 要素 | 採用内容 |
|---|---|
| モデル | `intfloat/multilingual-e5-small` |
| 次元数 | 384 |
| 保存先 | `articles.embedding vector(384)` |
| 距離指標 | cosine distance |
| DB演算子 | `<=>` |
| インデックス | HNSW `vector_cosine_ops` |

E5 系モデルの推奨に合わせ、クエリは `query: {q}`、記事本文は `passage: {title}\n{content}` の形式で embedding 化する。

### 3.2 スコア

pgvector の `<=>` は cosine distance を返すため、API の score は以下で類似度に変換する。

```text
semantic_score = 1 - cosine_distance
```

`embedding` が未生成の記事は semantic/hybrid の semantic 候補から除外する。作成・更新・CSV取込時に embedding 生成へ失敗した場合はエラーとして扱い、古い embedding を使った検索結果を返さない。

## 4. ハイブリッド検索

### 4.1 採用方式

ハイブリッド検索は RRF（Reciprocal Rank Fusion）を採用する。keyword と semantic はスコア分布が異なるため、スコアを直接合算せず順位ベースで融合する。

```text
rrf_score = 1 / (60 + keyword_rank) + 1 / (60 + semantic_rank)
```

片方の候補にしか存在しない記事は、存在する側の順位だけで RRF スコアを計算する。

### 4.2 候補取得

| 段階 | 内容 |
|---|---|
| 1 | categoryId、authorId で対象記事を絞り込む |
| 2 | keyword 上位候補を `limit * 5` 件取得する |
| 3 | semantic 上位候補を `limit * 5` 件取得する |
| 4 | article id 単位で候補を統合し、RRF スコアを計算する |
| 5 | RRF スコア降順でページングして返す |

`limit * 5` は、ページング後の結果品質を確保するための初期値とする。実装時に性能問題が出た場合は設定値として調整可能にする。

## 5. API・画面表示

### 5.1 API

検索は `GET /api/articles` で扱い、記事検索専用の別エンドポイントは作成しない。

| パラメータ | 説明 |
|---|---|
| q | 検索語または自然文クエリ |
| searchMode | `keyword`、`semantic`、`hybrid` |
| categoryId | カテゴリ絞り込み |
| authorId | 著者絞り込み |
| sort | `q` 指定時は `relevance` を既定値とする |

検索結果の `score` は mode ごとの関連度であり、異なる mode 間で絶対比較しない。

### 5.2 画面表示

| searchMode | 表示方針 |
|---|---|
| keyword | タイトル・本文の一致箇所に近いスニペットを表示する |
| semantic | 意味的に近い本文冒頭または検索意図に近いスニペットと類似度スコアを表示する |
| hybrid | キーワード一致と意味的類似の両方を考慮した順位として表示する |

## 6. 後続拡張

チャンク検索を実装する場合は、`article_chunks` テーブルを追加し、記事本文を段落またはトークン長で分割して chunk embedding を保存する。検索結果は chunk の類似度で候補を取得し、最終的には article id 単位に集約する。

今回の設計では実装対象外のため、DB設計・API仕様にはチャンク用テーブルや chunk id を追加しない。
