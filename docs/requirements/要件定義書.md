# 機能要件書

## 1. システム概要

### 1.1 プロジェクト概要

TechInsight は、技術記事データを管理・検索する AI 搭載型ナレッジベースである。
ユーザーは技術記事を登録、閲覧、編集、削除できる。キーワード検索とセマンティック検索により、具体的な技術用語を知らない場合でも、自然言語で目的に近い記事を探せるようにする。

本システムはコーディング試験の提出物として、評価者のローカル環境で `docker compose up` により再現できる Web アプリケーションとして構築する。評価者は外部 API キーを持っていない前提とし、AI/ML 機能は API キー不要のローカル実行可能な embedding model を採用する。

### 1.2 解決したい課題

- 技術記事が増えると、単純なキーワード検索だけでは目的の記事を見つけにくい。
- ユーザーが正確な技術用語を知らなくても、自然言語の意図に近い記事を検索できるようにする。
- CRUD、検索、AI 活用、Docker による再現性を備えた実用的なナレッジベースを構築する。
- 1,000 件の CSV データを扱い、将来的に 1 万件程度へ増えても破綻しない DB・検索設計にする。

### 1.3 スコープ

#### MVP スコープ

- Docker Compose による DB、Backend API、Frontend の一発起動
- PostgreSQL 起動、DB マイグレーション、pgvector 拡張の有効化
- `docs/articles.csv` の初期インポート
- 記事一覧取得、記事詳細取得、記事作成、記事更新、記事削除
- キーワード検索、セマンティック検索
- 記事一覧・検索画面、記事詳細表示、記事作成・編集・削除 UI
- README、簡易 DB 設計書、簡易 API 設計書、実装説明資料

#### 追加実装スコープ

- pgvector を利用したベクトル検索
- キーワード検索とセマンティック検索を組み合わせたハイブリッド検索
- 記事本文のチャンク分割、チャンク単位のセマンティック検索
- タグ管理・タグによる絞り込み
- 検索結果への類似度スコアと該当箇所スニペットの表示
- 記事詳細画面での関連記事表示
- 記事更新時の embedding 再生成
- 検索モード切り替え UI
- ローディング、エラー、空状態の UX 改善

#### 対象外スコープ

- ユーザー認証・認可
- 本番クラウド環境へのデプロイ
- 外部有料 API に依存する機能
- 複数ユーザーによる権限管理
- 本格的な監査ログ
- CI/CD の完全構築

### 1.4 技術構成

| 項目 | 採用技術 | 採用理由 |
|---|---|---|
| Backend | Python / FastAPI | 型定義と OpenAPI 生成に対応し、CRUD API と検索 API を短期間で実装しやすい |
| Frontend | Next.js / TypeScript / React | 既存テンプレートと整合し、画面単位の実装と型安全な API 連携がしやすい |
| Database | PostgreSQL | 構造化データ、全文検索、インデックス設計に強く、1 万件規模に十分対応できる |
| Vector Search | pgvector | PostgreSQL 上で embedding を保存し、距離演算と vector index を利用できる |
| AI/ML | ローカル embedding model | 評価者が API キーなしで再現できる |
| Infrastructure | Docker Compose | DB、API、Frontend を評価者環境でワンコマンド起動できる |
| Migration | Alembic | SQLAlchemy と相性がよく、DB スキーマ変更を管理しやすい |
| ORM | SQLAlchemy | 既存 Backend 構成と整合し、Repository 層で DB 実装を分離しやすい |
| Package manager | npm | 既存 Frontend の `package-lock.json` と整合する |

Embedding model は、API キー不要かつ日本語・英語混在の技術記事に対応しやすい軽量多言語モデルを採用する。候補は `intfloat/multilingual-e5-small` とし、Docker 初回起動時のモデル取得負荷が大きい場合は README に所要時間とトラブルシューティングを記載する。

### 2.1 主要ユースケース

| ユースケース | 目的 | 主な操作 | 正常系 | 異常系 |
|---|---|---|---|---|
| 記事一覧を閲覧する | 登録済み記事を確認する | 一覧画面を開く | ページングされた記事一覧が表示される | API エラー時にエラー表示を行う |
| キーワードで記事を検索する | 用語一致で探す | 検索語を入力して keyword モードで検索 | タイトル・本文に一致する記事が表示される | 0 件時に空状態を表示する |
| 自然言語で記事をセマンティック検索する | 意味的に近い記事を探す | 自然文を入力して semantic モードで検索 | 類似度順に記事が表示される | モデルロード失敗時に原因を表示する |
| 検索モードを切り替える | 検索方式を選ぶ | UI 上の検索モードを変更する | 選択した検索方式で再検索できる | 不正な mode は API で 400 を返す |
| 記事詳細を確認する | 本文全体を読む | 一覧または検索結果から記事を選択する | 詳細モーダルに本文が表示される | 対象 ID がない場合は 404 を表示する |
| 記事を新規作成する | ナレッジを追加する | 作成フォームを送信する | 記事が保存され、検索用データが作成される | 入力不備時にバリデーションエラーを表示する |
| 記事を編集する | 記事内容を更新する | 編集フォームを送信する | 記事と検索用データが更新される | 競合または対象なしの場合にエラーを表示する |
| 記事を削除する | 不要な記事を消す | 削除確認後に実行する | 記事が削除され、一覧から消える | 対象なしの場合は 404 を表示する |
| CSV から初期データを取り込む | 初回データを用意する | `docker compose up` を実行する | マイグレーション後に未投入データが登録される | CSV 不在時にログで原因を示す |
| 関連記事を確認する | 近い記事へ移動する | 記事詳細を開く | 類似記事が表示される | 関連記事がない場合は空状態を表示する |

## 3. 機能要件

### 3.1 記事管理機能

#### FR-001: 記事一覧取得

- **説明**: 登録済み記事を公開日時の降順で一覧取得する。
- **入力**: `page`、`limit`、任意のソート条件
- **出力**: 記事 ID、タイトル、本文抜粋、著者、カテゴリ、公開日時、総件数
- **ビジネスルール**: 1 万件程度のデータを想定し、必ずページネーションを行う。記事削除は物理削除で行うため、削除済み記事は DB に残らない。
- **受入基準**: 一覧画面で初期データの記事が表示され、ページ切り替えで追加記事を確認できる。

#### FR-002: 記事詳細取得

- **説明**: 指定された記事 ID の全文とメタデータを取得する。
- **入力**: 記事 ID
- **出力**: 記事 ID、タイトル、本文、著者、カテゴリ、公開日時
- **ビジネスルール**: 存在しない ID の場合は 404 を返す。
- **受入基準**: 一覧または検索結果から記事を選択すると、詳細モーダルで全文を確認できる。

#### FR-003: 記事作成

- **説明**: 新しい技術記事を登録する。
- **入力**: タイトル、本文、著者、カテゴリ、公開日時
- **出力**: 作成された記事情報
- **ビジネスルール**: タイトルと本文は必須とする。作成時に検索用データと embedding を生成する。
- **受入基準**: 作成フォームから記事を登録でき、登録後に一覧・検索対象へ反映される。

#### FR-004: 記事更新

- **説明**: 既存記事の内容を更新する。
- **入力**: 記事 ID、タイトル、本文、著者、カテゴリ、公開日時
- **出力**: 更新後の記事情報
- **ビジネスルール**: 本文またはタイトルが変わった場合は embedding を再生成する。存在しない ID の場合は 404 を返す。
- **受入基準**: 編集フォームから記事を更新でき、更新後の内容が詳細画面と検索結果に反映される。

#### FR-005: 記事削除

- **説明**: 既存記事を削除する。
- **入力**: 記事 ID
- **出力**: 削除結果
- **ビジネスルール**: 削除前に UI で確認ダイアログを表示する。削除フラグは持たせず、記事を物理削除する。
- **受入基準**: 削除後の記事は一覧、詳細、検索結果に表示されない。

#### FR-006: 入力バリデーション

- **説明**: 記事作成・更新時の入力値を検証する。
- **入力**: フォーム入力値または API リクエスト body
- **出力**: 正常時は保存結果、異常時はバリデーションエラー
- **ビジネスルール**: タイトルと本文は必須、文字列長は実用的な上限を設定する。Frontend と Backend の両方で検証する。
- **受入基準**: 未入力や不正値を送信した場合に、保存されず、ユーザーが修正箇所を理解できる。

### 3.2 検索機能

#### FR-007: キーワード検索

- **説明**: タイトル・本文を対象にキーワード一致で記事を検索する。
- **入力**: `query`、`mode=keyword`、`page`、`limit`
- **出力**: 記事 ID、タイトル、スニペット、公開日時、総件数
- **ビジネスルール**: PostgreSQL の index または全文検索 index を利用し、1 万件程度でも実用的な応答速度を維持する。
- **受入基準**: 技術用語を入力すると、該当語を含む記事が検索結果に表示される。

#### FR-008: セマンティック検索

- **説明**: 自然言語クエリと記事本文の embedding 類似度に基づき記事を検索する。
- **入力**: `query`、`mode=semantic`、`page`、`limit`
- **出力**: 記事 ID、タイトル、スニペット、類似度スコア、公開日時
- **ビジネスルール**: 検索クエリを embedding に変換し、article embedding との cosine similarity もしくは pgvector の距離演算で類似度を計算する。結果は類似度の高い順に返す。
- **受入基準**: 例として「API の応答が遅い原因を調べたい」と検索した場合、「PostgreSQL のクエリ最適化」「FastAPI の非同期処理」「キャッシュ戦略」など意味的に近い記事が返る。

#### FR-009: ハイブリッド検索

- **説明**: キーワード検索とセマンティック検索の結果を組み合わせる。
- **入力**: `query`、`mode=hybrid`、`page`、`limit`
- **出力**: 記事 ID、タイトル、スニペット、類似度スコア、キーワード一致情報
- **ビジネスルール**: MVP 完成後の追加実装とする。キーワード一致とベクトル類似度のスコアを統合して順位付けする。
- **受入基準**: 完全一致の記事と意味的に近い記事の両方が、納得感のある順序で表示される。

#### FR-010: 検索結果表示

- **説明**: 検索結果の理由が分かる情報を Frontend に表示する。
- **入力**: 検索 API のレスポンス
- **出力**: タイトル、スニペット、類似度スコア、検索モード、0 件表示
- **ビジネスルール**: MVP では基本結果表示を必須とし、類似度スコアとスニペットの改善は追加実装として扱う。
- **受入基準**: 検索結果が 0 件の場合、空状態が表示される。セマンティック検索では類似度の根拠を確認できる。

### 3.3 Embedding 機能

#### FR-011: 記事本文のベクトル化

- **説明**: 記事本文を embedding に変換し、検索用データとして保存する。
- **入力**: 記事タイトル、記事本文
- **出力**: embedding vector
- **ビジネスルール**: API キー不要のローカルモデルを使う。モデルは Docker 環境で動作し、評価者が追加契約なしで再現できるものにする。
- **受入基準**: CSV インポート、記事作成、記事更新時に embedding が生成または再生成される。

#### FR-012: チャンク分割

- **説明**: 長い記事本文を検索しやすい単位に分割する。
- **入力**: 記事本文
- **出力**: chunk 一覧、chunk embedding
- **ビジネスルール**: MVP 完成後の追加実装とする。記事本文を一定文字数または文単位で分割し、chunk ごとに embedding を保持する。
- **受入基準**: 検索結果のスニペットが該当 chunk から生成され、長文記事でも関連箇所を見つけやすい。

### 3.4 初期データ投入機能

#### FR-013: CSV インポート

- **説明**: 提供された `docs/articles.csv` 1,000 件を PostgreSQL に取り込む。
- **入力**: `docs/articles.csv`
- **出力**: `articles` レコード
- **ビジネスルール**: マイグレーション後に初期投入を行う。CSV のヘッダーは `id,title,content,author,category,published_at` とする。`content` は DB の本文カラム `content` に取り込む。`category` は `categories`、`author` は `authors` に登録し、記事から外部キーで参照する。再実行時に重複投入されないよう、`source_article_id` の一意制約で冪等性を担保する。
- **受入基準**: 初回 `docker compose up` 後に記事一覧で CSV データを確認できる。再起動しても同じ記事が重複しない。

### 3.5 フロントエンド機能

#### FR-014: 記事一覧・検索画面

- **説明**: 記事一覧、検索フォーム、検索モード切り替えを提供する。
- **入力**: 検索語、検索モード、ページ操作
- **出力**: 記事一覧、検索結果、ローディング、エラー、空状態
- **ビジネスルール**: 一覧と検索は同一画面で扱い、利用者が検索結果から詳細表示や編集操作へ移れるようにする。
- **受入基準**: 初期表示、検索、ページング、詳細表示への遷移が画面上で完結する。

#### FR-015: 記事詳細・作成・編集・削除 UI

- **説明**: 記事詳細モーダル、記事作成フォーム、記事編集フォーム、削除確認ダイアログを提供する。
- **入力**: ユーザー操作、フォーム入力
- **出力**: 詳細情報、保存結果、削除結果、成功・失敗フィードバック
- **ビジネスルール**: 認証がないため、同一ユーザーが閲覧と管理操作を行う。破壊的操作は確認ダイアログを必須にする。
- **受入基準**: 画面上で記事の作成、編集、削除ができ、操作結果が一覧へ反映される。

### 3.6 DB 設計機能

#### FR-016: DB スキーマ

- **説明**: 記事データと検索用データを管理する DB スキーマを定義する。
- **入力**: マイグレーションファイル
- **出力**: `articles`、`categories`、`authors`
- **ビジネスルール**: 最低限、`articles` は `id`、`source_article_id`、`category_id`、`author_id`、`title`、`content`、`published_at`、`embedding` を持つ。`source_article_id` は `docs/articles.csv` の `id` を保持し、CSV 再投入時の冪等性担保に利用する。`docs/articles.csv` の `category` は `categories`、`author` は `authors` で管理する。MVP のセマンティック検索は記事単位の `embedding` で実現する。
- **受入基準**: Alembic により DB スキーマが作成され、アプリケーションから CRUD と検索に利用できる。

#### FR-017: DB インデックス

- **説明**: 一覧・検索・関連取得の性能を確保する index を定義する。
- **入力**: マイグレーションファイル
- **出力**: `articles.published_at`、`articles.title`、`articles.category_id`、`articles.author_id`、`articles.embedding`、必要に応じた全文検索 index
- **ビジネスルール**: 1 万件程度の記事データを想定し、検索 API ではページネーションと index を併用する。
- **受入基準**: 1 万件程度の想定データでも、一覧取得と検索 API が実用的な応答時間で動作する設計になっている。

### 3.7 API 設計機能

#### FR-018: API エンドポイント

- **説明**: CRUD、検索、関連記事、再インデックス用 API を提供する。
- **入力**: HTTP リクエスト
- **出力**: JSON レスポンス
- **ビジネスルール**: OpenAPI で確認できるよう FastAPI のスキーマを整備する。エラー時は一貫した形式でステータスコードとメッセージを返す。
- **受入基準**: Swagger UI または API 設計書から各 API の目的、リクエスト、レスポンス、エラーケースを確認できる。

| API | 目的 | 主なリクエスト | 主なレスポンス | エラーケース |
|---|---|---|---|---|
| `GET /api/articles` | 記事一覧取得 | `page`, `limit` | 記事一覧、総件数 | 不正なページ指定は 400 |
| `GET /api/articles/{id}` | 記事詳細取得 | 記事 ID | 記事詳細 | 対象なしは 404 |
| `POST /api/articles` | 記事作成 | タイトル、本文、著者、カテゴリ、公開日時 | 作成記事 | 入力不備は 422 |
| `PUT /api/articles/{id}` | 記事更新 | 記事 ID、更新項目 | 更新記事 | 対象なしは 404 |
| `DELETE /api/articles/{id}` | 記事削除 | 記事 ID | 削除結果 | 対象なしは 404 |
| `GET /api/articles` | 検索 | `q`, `searchMode`, `page`, `limit` | 検索結果 | 不正 mode は 400 |
| `GET /api/articles/{id}/related-articles` | 関連記事取得 | 記事 ID | 関連記事一覧 | 対象なしは 404 |
| `POST /api/reindex-jobs` | 再インデックス | 任意の対象指定 | 実行結果 | 処理失敗は 500 |

## 4. 非機能要件

### 4.1 パフォーマンス

- 1,000 件の CSV データを初期投入し、将来的に 1 万件程度の記事データを扱える設計にする。
- 一覧 API と検索 API はページネーションを必須とする。
- PostgreSQL に `published_at`、`title`、`category_id`、`author_id`、全文検索、vector 検索に必要な index を設定する。
- ベクトル検索には pgvector の index を利用する。
- チャンク単位検索は追加実装とし、MVP では記事本文全体の embedding を利用する。

### 4.2 セキュリティ

- 今回は認証・認可を対象外とするが、API 入力値のバリデーションは必須とする。
- SQL Injection を避けるため、SQLAlchemy の ORM または安全なパラメータバインディングを使用する。
- `.env.example` を用意し、秘密情報をリポジトリに含めない。
- 外部 API キーがないと動かない設計にしない。

### 4.3 可用性

- 評価対象はローカル実行であり、本番可用性やクラウド冗長化は対象外とする。
- `docker compose up` により DB、API、Frontend が依存順序を満たして起動する。
- DB 起動確認後にマイグレーションと初期データ投入を実行する。
- モデルロードや CSV 投入に失敗した場合はログで原因を確認できるようにする。

### 4.4 スケーラビリティ

- Backend / Frontend / Database / Docs を分離したディレクトリ構成にする。
- DB マイグレーションでスキーマ変更を管理する。
- API スキーマと型定義を明確にし、Frontend と Backend の連携ミスを減らす。
- 検索方式は keyword、semantic、hybrid を mode として拡張可能にする。
- MVP では記事単位 embedding に絞り、チャンク単位 embedding は追加実装時に拡張する。

### 4.5 再現性

- 評価者が `docker compose up` だけで DB、API、Frontend を起動できる。
- API キーなしでも CRUD、キーワード検索、セマンティック検索が動作する。
- README に起動手順、動作確認手順、初回モデルロード、CSV 配置、トラブルシューティングを記載する。

### 4.6 保守性・チーム開発性

- Backend は責務分離を意識し、API 層、Application 層、Domain 層、Infrastructure 層を分離する。
- Frontend は画面、feature、shared component、API client を分離する。
- 命名規則、Lint / Format 方針、テスト方針を README または開発ドキュメントに記載する。
- 簡易 DB 設計書と簡易 API 設計書を整備し、チーム開発時の認識齟齬を減らす。

### 4.7 UX

- 検索結果の理由が分かる UI にする。
- 類似度スコア、スニペット、検索モードを表示できるようにする。
- 操作成功・失敗が分かるフィードバックを表示する。
- ローディング、エラー、空状態を用意する。
- 作成・編集フォームでは入力エラーを項目単位で表示する。

## 5. 制約事項

### 5.1 対象外

- ユーザー認証・認可は実装しない。
- 本番クラウド環境へのデプロイは行わない。
- 外部有料 API に依存する機能は実装しない。
- 複数ユーザーの権限管理は実装しない。
- 本格的な監査ログは実装しない。
- CI/CD の完全構築は必須としない。

### 5.2 リスクと対策

| リスク | 対策 |
|---|---|
| ローカル embedding model の初回ダウンロードが重い | README に初回起動時の所要時間と事前取得手順を記載する |
| Docker 環境でモデルロードに時間がかかる | 起動ログを明確にし、必要に応じて lightweight model を採用する |
| pgvector の設定ミス | 初回 DB 起動時に `CREATE EXTENSION IF NOT EXISTS vector;` を実行する |
| CSV 再投入時の重複 | `source_article_id` の一意制約で冪等な import にする |
| 記事更新時に embedding が古くなる | 作成・更新処理の保存フローで embedding 再生成を行う |
| セマンティック検索の精度が評価者に伝わりにくい | README に検索例を記載し、UI に score と snippet を表示する |
| 外部 API キーがないと動かない設計になる | API キー不要のローカルモデルを採用し、外部 API は使わない |
